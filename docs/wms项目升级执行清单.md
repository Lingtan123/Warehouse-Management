# WMS 项目升级执行清单

## 1. 文档用途

这份文档只做一件事：

指导你把当前 `WarehouseManagement` 项目逐步升级成一个更适合秋招面试的 Java 后端项目。

你后续直接对照这份清单改项目即可。

如果某一步卡住了，直接这样告诉我：

```text
步骤编号：2.2
修改的文件：wms/src/main/java/com/wms/...
我做了什么：...
现在的问题：...
我期望的结果：...
报错信息或现象：...
```

这样我就可以直接接着帮你，不需要你重新解释背景。

## 2. 总目标

把你现在这个项目，从“基础仓库管理 CRUD 项目”，升级成“能讲工程化、能讲安全、能讲一致性、能讲缓存和异步”的秋招项目。

最终目标技术栈：

- 后端基础：`Spring Boot + MyBatis-Plus + MySQL`
- 安全认证：`JWT + BCrypt`
- 缓存：`Redis`
- 消息队列：`RabbitMQ`
- 部署：`Docker + Docker Compose`
- 测试：`Spring Boot Test`
- 可选增强：`轻量 Agent 模块`

注意：

- 先补全项目，再考虑 Agent
- 先做单体增强，不急着拆微服务
- 先把项目做扎实，不追求技术名词很多

## 3. 升级顺序

严格按这个顺序来：

1. 项目结构整理
2. 登录鉴权升级
3. 库存业务升级
4. 数据库约束和索引
5. Redis 接入
6. RabbitMQ 接入
7. Docker 部署
8. 测试补全
9. 文档和简历包装
10. 可选 Agent 增强

## 4. 第一阶段：项目结构整理

### 1.1 先确认当前项目能正常运行

目的：

建立一个稳定基线，避免后面改着改着不知道哪里坏了。

你要做的事：

- 启动后端 `wms`
- 启动前端 `wms-web`
- 手动测试一次登录
- 手动测试一次商品或库存操作

你主要看这些位置：

- `wms/pom.xml`
- `wms/src/main/resources/`
- `wms-web/src/`

完成标准：

- 后端能启动
- 前端能启动
- 登录功能能走通
- 至少一个业务操作能成功

---

### 1.2 把业务逻辑从 Controller 挪到 Service

目的：

让代码结构更像正式项目，而不是课程作业。

当前优先改的文件：

- `wms/src/main/java/com/wms/controller/RecordController.java`

你要做的事：

- 把出库、入库、库存计算这些业务逻辑从 `RecordController` 中抽出去
- 放到 `service` 层处理
- Controller 只保留“收参数、调 service、返回结果”

建议新增或改造的位置：

- `wms/src/main/java/com/wms/service/IRecordService.java`
- `wms/src/main/java/com/wms/service/impl/RecordServiceImpl.java`

建议新增的方法：

- `stockIn(...)`
- `stockOut(...)`
- `saveInventoryOperation(...)`

完成标准：

- Controller 中不再直接写库存加减逻辑
- 业务规则集中到 Service

---

### 1.3 把 `System.out.println` 改成日志

目的：

让项目更工程化，也更方便排查问题。

你要加的技术：

- `Slf4j`

优先改这些文件：

- `wms/src/main/java/com/wms/controller/UserController.java`
- `wms/src/main/java/com/wms/controller/RecordController.java`

你要做的事：

- 去掉 `System.out.println`
- 改成 `log.info(...)`
- 异常时用 `log.error(...)`

完成标准：

- 核心接口里不再依赖 `System.out.println`

---

### 1.4 引入 DTO，不直接拿实体类接请求

目的：

减少前后端请求和数据库实体的耦合。

你要加的内容：

- 新建 `dto` 包

建议新增位置：

- `wms/src/main/java/com/wms/dto/`

第一批建议新增类：

- `LoginRequestDTO`
- `InventoryOperateDTO`
- `UserQueryDTO`

你要做的事：

- 登录接口不要直接接 `User`
- 出入库接口不要直接接原始实体
- 查询接口把查询条件单独封装

完成标准：

- 核心接口入参改成 DTO

---

### 1.5 增加统一异常处理和参数校验

目的：

这是最容易提升项目质量的一步。

你要加的技术：

- `Bean Validation`
- `@Valid`
- 全局异常处理

建议新增位置：

- `wms/src/main/java/com/wms/exception/`

建议新增类：

- `BusinessException`
- `GlobalExceptionHandler`

你要做的事：

- DTO 上加注解，比如 `@NotBlank`、`@NotNull`、`@Min`
- Controller 方法参数加 `@Valid`
- 所有业务异常通过统一异常处理返回

完成标准：

- 参数错了能返回统一格式错误信息
- 业务报错不再直接抛到前端

## 5. 第二阶段：登录鉴权升级

### 2.1 密码加密

目的：

现在如果还是明文密码查库，面试里会比较吃亏。

你要加的技术：

- `BCryptPasswordEncoder`

当前重点文件：

- `wms/src/main/java/com/wms/controller/UserController.java`

你要做的事：

- 注册或保存用户时，把密码加密后再存库
- 登录时不再用“账号 + 明文密码直接查库”
- 改成“先查账号，再用 BCrypt 比较密码”

你可能要改的位置：

- `UserController`
- `IUserService`
- `UserServiceImpl`

完成标准：

- 数据库中密码不再是明文
- 登录能正确校验加密密码

---

### 2.2 JWT 登录认证

目的：

让你的项目有标准的登录态管理。

你要加的技术：

- `JWT`

建议新增位置：

- `wms/src/main/java/com/wms/util/`
- `wms/src/main/java/com/wms/config/`
- `wms/src/main/java/com/wms/interceptor/` 或 `filter/`

建议新增类：

- `JwtUtil`
- `LoginInterceptor` 或 `JwtAuthenticationFilter`

你要做的事：

- 登录成功后返回 token
- token 中带上用户 id、角色 id
- 除登录接口外，其他需要保护的接口都校验 token

完成标准：

- 登录接口返回 token
- 不带 token 无法访问受保护接口

---

### 2.3 后端角色权限控制

目的：

现在如果只是前端菜单控制，不够，后端也要拦。

你要做的事：

- 区分管理员和普通用户权限
- 普通用户限制高危操作
- 后端接口内部做角色判断

建议保护的接口：

- 用户管理
- 商品新增、删除、修改
- 仓库新增、删除、修改
- 出入库操作

完成标准：

- 后端真正拦住越权请求

---

### 2.4 增加操作审计基础能力

目的：

让项目更像真实业务系统。

你要做的事：

- 记录谁做了什么操作
- 至少记录登录、出库、入库、删除

实现方式先简单一点即可：

- 可以先记日志
- 后面也可以异步落库

完成标准：

- 关键操作可追溯到用户和时间

## 6. 第三阶段：库存业务升级

这是你整个项目最值钱的一部分。

### 3.1 统一库存操作入口

目的：

所有库存变更都从一处走，方便后续加事务、加锁、加日志。

你要做的事：

- 统一封装库存操作方法
- 不允许多个 Controller 各自改库存

建议集中位置：

- `RecordServiceImpl`
- 或单独新建 `InventoryService`

完成标准：

- 所有库存变更都走统一 service 方法

---

### 3.2 给出入库操作加事务

目的：

保证“改库存”和“写流水”要么都成功，要么都失败。

你要加的技术：

- `@Transactional`

你要做的事：

- 入库：库存增加 + 记录流水 放在一个事务里
- 出库：库存减少 + 记录流水 放在一个事务里

重点位置：

- `wms/src/main/java/com/wms/service/impl/RecordServiceImpl.java`

完成标准：

- 中间有一步失败，前面的数据库修改会回滚

---

### 3.3 防止库存扣成负数

目的：

这是仓储项目必须有的业务约束。

你要做的事：

- 出库前先查当前库存
- 如果库存不足，直接报业务异常

建议处理方式：

- 抛 `BusinessException`
- 由 `GlobalExceptionHandler` 统一返回错误

完成标准：

- 库存不足时，系统不允许出库

---

### 3.4 增加乐观锁

目的：

面试里可以讲并发下如何避免库存覆盖。

你要加的技术：

- `MyBatis-Plus 乐观锁`

你要改的地方：

- 商品表或库存表加 `version` 字段
- 对应实体类加版本字段
- 配置 MyBatis-Plus 乐观锁插件

重点文件：

- `wms/src/main/java/com/wms/entity/Goods.java`
- `wms/src/main/java/com/wms/common/MybatisPlusConfig.java`

完成标准：

- 并发修改库存时，不会无声覆盖

---

### 3.5 考虑幂等性

目的：

防止用户重复提交导致重复扣库存。

第一版不用做太复杂，你可以先做：

- 前端按钮防重复点击
- 后端通过请求号或业务号做简单防重

完成标准：

- 同一笔请求重复提交时，系统有拦截或去重思路

---
